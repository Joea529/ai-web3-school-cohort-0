## 第一性原理

模型只能基于它看见的上下文行动；系统必须决定什么能进上下文、带着什么身份进去、过期后怎么退出。

Context 不是文本拼接，是信息治理——为每类信息标注来源、时效、权限和可信度。否则模型会把"用户说的""网页写的""链上查到的""系统规定的"混在同一层处理。

## 4 个知识节点

### 1. Context Window
模型一次请求的最大上下文范围。窗口越大 ≠ 越靠谱——长上下文的真问题是"看见了但没抓住重点"。实际产品里关键状态直接给，长文档按需取，低可信内容隔离标注。

### 2. Context Engineering
设计上下文进入模型的方式：选数据源、排序、裁剪、标注来源、隔离不可信内容。稳定的 Agent 上下文应包含七个维度：
当前任务状态 + 工具返回结果 + 相关日志或证据 + 可信数据来源 + 外部检查结果 + 用户原始意图 + 系统禁止事项和输出 schema。

### 3. Memory
跨请求保留的信息（偏好、历史任务、常用钱包、项目配置）。好用但有隐性风险——记住"常用地址"可能变成默认收款地址，记住"风险偏好高"可能放松确认。**Memory 不能替代实时授权**——所有涉及身份、权限、资产或外部副作用的记忆，必须重新绑定当前会话和授权。

### 4. Knowledge Base
系统可检索的外部知识库。解决模型知识过期问题，但不会自动保证正确。知识库需要维护：来源 URL、更新时间、适用版本、是否官方、是否需要人工复核。否则 RAG 只会把旧错误更稳定地送进模型。

## AI × Web3 中的位置

Context 是连接模型和现实系统的入口。AI 负责推理，Web3 负责状态和结算——context 决定模型看到的是用户幻想、过期文档，还是可验证的链上事实。

靠谱的 Agent 上下文分五层：
- **指令层**：系统规则、工具规则、禁止事项
- **任务层**：用户目标、本次会话参数
- **事实层**：链上状态、工具结果、simulation
- **知识层**：文档、标准、论坛、历史案例
- **记忆层**：用户偏好和项目配置

层级越清楚，权限控制、prompt injection 防护和审计越容易做。

## 最小实践（待做）

为"钱包授权检查 Agent"设计 context spec。用户问"这个 dApp 要我 approve，可以签吗？"时，模型需要：
- chain id + 当前区块
- token 合约、spender 地址、approve 数量
- 当前 allowance + 余额
- spender 是否在可信列表
- simulation / 静态检查结果
- dApp 页面说明（标记为不可信外部内容）
- 用户本次意图

并标注哪些字段必须实时查询，哪些可来自缓存，哪些不能被模型当成事实。
