## 第一性原理

Vibe Coding 不是"不用动脑写代码"，而是把认知资源从语法和实现细节中释放出来，集中到"要做什么"和"怎么验证"上。

核心转变：从"我写代码让机器执行"变成"我描述意图，AI 生成实现，我负责验证和迭代"。人的角色从打字员变成导演。

## 6 个知识节点

### 1. 什么是 Vibe Coding
Andrej Karpathy 2025 年提出的概念：用自然语言描述需求，让 AI 生成代码，开发者通过迭代 prompt 调整结果，而非直接修改代码。状态是"完全沉浸在氛围中，忘记代码本身的存在"。

这不是偷懒，是交互范式的迁移——就像从汇编到高级语言、从命令行到 GUI，每次迁移都释放了一层认知负担。

### 2. 核心工作流
```
需求描述 → AI 生成代码 → 运行验证 → 发现问题 → 自然语言反馈 → AI 修改 → 再验证
```

关键不是第一次生成多好，而是反馈循环多快。越短的迭代周期，越像"对话"而非"开发"。理想的 vibe coding 状态是：你说一句话，AI 改完，你马上看到结果，然后说下一句话。

### 3. 适合什么场景
- 原型验证：快速出可运行的 demo，验证想法可行性
- UI/前端：描述布局和交互，AI 生成组件
- 脚本和工具：一次性数据处理、自动化脚本
- 学习探索：通过"让 AI 实现某个概念"来反向理解该概念

不适合：核心算法（需要精确性和可解释性）、安全关键系统、需要深度性能优化的场景。

### 4. 常用工具
- **Cursor / Copilot**：IDE 内嵌，适合边写边补全
- **Claude Code / Codex CLI**：终端 Agent，适合多文件、多步骤任务
- **Bolt / v0 / Replit Agent**：从描述直接生成完整应用
- **Hermes Agent**：我们正在用的——不只是 vibe coding 工具，更是 vibe coding 的"元工具"（可以编排多个 AI agent 协作）

### 5. 与前面章节的关系
Vibe Coding 不是孤立概念，它依赖前六章的全部积累：

| 依赖章节 | 在 Vibe Coding 中的作用 |
|----------|------------------------|
| LLM | 底层模型能力决定生成质量上限 |
| Prompt | 描述需求的精度直接影响生成结果 |
| Context | 给 AI 足够的上下文（文件、文档、规范）才能生成准确代码 |
| RAG | 让 AI 能检索项目文档和代码库，理解现有架构 |
| Agent | Vibe Coding 工具本质上是 Coding Agent——能调用工具、读写文件、执行命令 |
| Frameworks | 框架决定了 AI 生成代码的结构和质量 |

Vibe Coding 是前面六章的自然汇聚点。

### 6. 陷阱和边界
- **幻觉代码**：AI 可能凭空发明不存在的 API 或库——必须运行验证
- **过度生成**：简单改一行的事，AI 可能重写整个文件——需要约束 scope
- **上下文丢失**：长对话中 AI 可能忘记之前的设计决策——需要显式记录架构上下文
- **安全风险**：不要把生产环境密钥、用户数据放进 prompt
- **不要跳过理解**：Vibe Coding 不等于不学习。看不懂 AI 生成的代码时，停下来搞懂再看下一步

## AI × Web3 中的位置

Vibe Coding 在 AI × Web3 场景中有特殊价值：

Web3 开发门槛高——钱包交互、合约调用、gas 计算、链上状态管理——每个都是新手的墙。Vibe Coding 降低实验成本：不用先成为 Solidity 专家就能做原型验证。

但也带来新风险：AI 生成的合约代码可能有漏洞，生成的交易逻辑可能错误处理边界条件。规则不变：生成→审查→测试→审查→部署。Vibe Coding 加速"生成"和"测试"，但不替代"审查"。

## 最小实践（待做）

用 vibe coding 方式完成同一个任务，对比三种工具：

**任务**：写一个命令行工具，输入 GitHub 用户名，输出该用户的公开 repo 列表（名称 + 星数 + 语言）

用三种方式各实现一次，计时并记录：
1. **纯手写**：从零写 Python 脚本
2. **Cursor/Copilot**：IDE 内补全辅助
3. **Claude Code / Hermes**：一句话描述，AI 完成全部

对比维度：完成时间、代码质量、错误数量、调试时间、成就感/挫败感。

💡 3 个核心理解：
1. Vibe Coding 是把认知资源从"怎么写"迁移到"做什么"和"怎么验证"——人的角色从打字员变成导演，迭代速度决定效率上限

🎯 最大收获：Vibe Coding 这个词最妙的地方在"vibe"——它不是偷懒，是进入一种更高效的认知状态。就像熟练的开发者不再想"分号要不要加"，vibe coder 不再想"这个 API 的参数顺序是什么"。每一层自动化都释放一层认知负担，让你思考更高维度的问题。这也解释了为什么前六章不是孤立的——它们就是 vibe coding 的认知基础设施。

❓ 如果 Vibe Coding 的终局是"所有人都能写软件"，那开发者的核心价值会迁移到哪里？我猜是三个方向：系统设计（知道做什么）、验证判断（知道对不对）、领域知识（知道为什么）。代码生成本身会变成 commodity。
