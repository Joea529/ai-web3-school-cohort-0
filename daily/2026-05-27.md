## 第一性原理

框架是系统边界的表达，不是智能本身。先理解工作流，再决定用不用框架。

很多失败的 AI 项目不是输在没有框架，而是先引入框架，再让产品逻辑迁就框架。更稳的方式是先画清楚输入、状态、工具、输出、评估和失败路径，再看哪些部分值得交给框架。

三条原则：
- 简单链路先保持简单 — 一次模型调用 + 一次检索 + 一次格式化输出，不一定需要 Agent 框架
- 长流程需要显式状态 — 多步任务、工具调用、确认、恢复，应该有可查询的 state，不是只靠聊天历史
- 框架要能退出 — 如果很难换模型、换向量库、换部署方式，长期成本很高

## 6 个知识节点

### 1. LangChain
最常见的 LLM 应用框架，覆盖模型接入、prompt、工具调用、retriever、agent、output parser。适合快速接模型与外部系统。但"抽象太早"是常见坑——没搞清工作流就套 chain 和 agent，排查变难。LangChain 更像组件库，适合组合能力，不适合替你决定产品边界。

### 2. LangGraph
偏向工作流和状态机。把 Agent / 多步骤任务表示成 graph：节点执行动作，边控制流，state 记录过程。判断标准：只要开始关心"任务走到哪一步、能否恢复、失败后从哪里继续"，就该考虑 graph / state machine。问一次答一次不需要。

### 3. OpenAI Agents SDK
构建带工具、handoff、guardrails、tracing 的 Agent 应用。关键仍然是边界：SDK 执行流程，你定义哪些工具可用、哪些动作需确认、什么输出算失败。

### 4. DSPy
把 prompt / LM pipeline 写成可优化的程序。不是手工调 prompt，而是定义输入输出、模块和指标，用 optimizer 改进。关键启发：不要只靠感觉调 prompt，要让任务、数据和指标进入系统。前提是有明确数据集和评估指标。

### 5. Hermes
作为"面向工具调用和结构化输出的模型/agent 生态"来理解。核心提示：模型本身是否擅长 tool calling、JSON mode、long context、reasoning，也会影响系统设计。不要只看榜单分数，要看是否能稳定产生你需要的结构化输出。

### 6. Learning Agent
从反馈、日志、评估结果中改进。不是训练模型，可以是更新 prompt、调整 retriever、补充规则、改进测试集。最稳的流程：记录失败 → 标注原因 → 加入 eval set → 修改配置 → 测试通过后上线。学习能力要先进评估闭环，再进生产。

## 框架选择速查

| 场景 | 推荐方向 |
|------|----------|
| 快速原型、学习组件 | LangChain |
| 多步任务、有状态、需恢复 | LangGraph |
| OpenAI 生态、handoff、tracing | OpenAI Agents SDK |
| 有数据集、需自动优化 prompt | DSPy |
| 纯 API 调用、一次问答 | 不用框架 |

## AI × Web3 中的位置

Frameworks 负责把模型能力接到产品流程：读取上下文、调用工具、生成结构化动作、记录 trace、进入 eval。

但它不应该直接替代 Web3 侧的权限、签名、交易模拟和账户规则。框架可以组织 Agent，不能替用户承担资产风险。

稳的分工：
1. AI Framework 管 prompt、tools、state、eval 和 trace
2. Web3 基础设施管账户、签名、合约、交易和链上状态
3. 产品层定义用户目标、权限边界、确认流程和失败处理

## 最小实践（待做）

同一个任务两种实现，对比四件事：
- **直接调 API**：用户问题 + 检索结果 + JSON 输出
- **用框架**：同样输入输出 + tool schema + trace + 失败重试 + 测试样本

对比：哪个更容易读懂？加工具？定位错误？写回归测试？
重点不是选冠军，是看清框架到底帮了什么。
