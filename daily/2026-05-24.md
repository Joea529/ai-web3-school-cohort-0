## 第一性原理

Agent 不是自主权本身，而是被约束的执行循环。目标、工具、状态、权限和停止条件缺一不可。

一个 Agent 如果只有模型和工具，就像一个会说话的脚本执行器。真正可用的 Agent 必须知道自己能做什么、不能做什么、做完如何验证、失败如何停止、谁能审计它做过的事。

## 5 个知识节点

### 1. Tool Use
工具让 Agent 从「会回答」变成「能做事」。但工具也让 Prompt Injection 和模型误判变得危险——能读网页 vs 能写入系统，后果完全不同。工具设计必须明确：输入 schema、权限范围、是否只读、是否产生副作用、调用前后记录、哪些需要人工确认。

### 2. Planning
把目标拆成步骤。计划只是候选路线，不是授权。好的 plan 暴露：每一步用什么工具、读还是写、是否可自动执行、哪些需确认、失败能否重试、如何验证完成。

### 3. State
Agent 当前任务状态（目标、已完成步骤、工具返回、错误、预算、确认记录、最终输出）。不能只放在 prompt 历史里——生产系统需要可查询、可恢复、可审计的 state。

### 4. Reflection
Agent 检查自己的中间结果并修正。可提高质量但不能替代外部验证——模型可能给自己的错误找理由。写入/授权/支付类动作，reflection 只是辅助诊断，不是最终安全判断。

### 5. Multi-Agent
多 Agent 分工协作。会放大协调问题：上下文丢失、责任模糊、错误被当成事实、工具权限扩散。做之前先问：多个 Agent 真的减少复杂度了吗？

## AI × Web3 中的位置

Agent 位在模型能力和链上执行之间。稳的架构通常是：
1. 用户提出目标和约束
2. Agent 读取上下文并生成计划
3. 系统拆成只读步骤和候选写入步骤
4. 只读自动执行，写入进入 policy 检查
5. Simulation 展示链上影响
6. 用户确认高风险动作
7. Wallet / Smart Account 执行
8. 日志记录每一步和最终状态

> 最危险的设计：Agent 同时拥有模糊目标、广泛工具、长期记忆和大额资产权限。

## 最小实践（待做）

做一个「DAO 提案研究 Agent」的最小版本——只能执行只读动作：
- 读取提案正文
- 检索论坛讨论
- 总结支持和反对理由
- 标记缺失信息
- 输出投票前检查清单

不要让它直接投票。要求在输出里明确：用了哪些来源、哪些结论证据不足、是否发现治理或资金风险、人工检查清单。
