## 第一性原理

RAG 的可靠性取决于证据链，不取决于向量库这个名词。

一个 RAG 系统至少有三次判断：文档怎么切（Chunking），查询时取哪些内容（Retriever），生成时如何引用和拒答（Citation）。任何一层做错，模型都会拿着错误材料说得很顺。

检索结果不是事实：它只是候选证据，仍要看来源、时间、版本和适用范围。找不到证据时，系统应该说"不确定"，而不是让模型补全。

## 5 个知识节点

### 1. Chunking
把长文档切成可检索片段。切太小语义断裂，切太大噪声多。技术文档不能按固定字数切——函数说明、参数表、风险提示跨段落出现，应按结构切：标题、API endpoint、函数说明、FAQ、审计记录。每个 chunk 保留来源 URL、标题路径、更新时间和版本。

### 2. Vector DB
存 embedding + metadata，按相似度检索。但向量相似 ≠ 答案正确——旧 SDK 文档可能和问题高度相似却已不适用。metadata 必须包含：来源、版本、chain、更新时间、可信等级、是否废弃。检索时先过滤再排序。

### 3. Retriever
根据用户问题取回候选材料。好的 retriever 不能只看语义相似度，还要知道用户问的是哪个产品、哪个版本、哪段时间、官方还是社区。实用判断：如果问题里有版本/环境/时间/地址，就不该只做纯向量搜索。

### 4. Rerank
初步检索后重新排序，把更相关、更可信、更完整的内容排到前面。增加延迟和成本，面向资产风险或开发者文档值得加，小型 FAQ 未必需要。

### 5. Citation
把答案里的关键结论连接回来源——不是装饰，是用户验证的入口。至少说明：来自哪份文档、是否官方、版本/更新时间、哪些是模型归纳、哪些缺乏证据。

## AI × Web3 中的位置

RAG 位在 Knowledge Base 和模型之间。帮 Agent 查资料、补上下文、引用证据，但不负责最终执行。

常见应用：协议文档问答、合约接口解释、治理提案摘要、审计报告检索、SDK/API Copilot、交易解释。

**关键约束**：RAG 结果要影响链上动作时，还需接 simulation、policy 和 human check。文档说某函数"可以调用"，不等于当前用户"应该签名"。

## 最小实践（待做）

做一个"协议文档 RAG 问答"的最小版本：
- 选一个官方文档站，抓 10-20 篇页面，按标题结构切 chunk，存入向量库
- 测试三类问题：文档中存在的 API、不存在的要求拒答、旧版本要求核对
- 输出时强制区分：answer / sources / uncertainties / needs_version_check
- 重点在证据链，不是 UI
